# Release Publishing Checklist

Before publishing **zencontrol-cloud-mcp** to PyPI, complete these steps:

## Pre-Publishing Setup (One-Time)

### 1. PyPI Account & Token
- [ ] Create a [PyPI account](https://pypi.org/account/register/) if you don't have one
- [ ] Create an API token at https://pypi.org/manage/account/tokens/ with "Entire repository" scope
- [ ] Save the token (format: `pypi-...`)

### 2. GitHub Repository Setup
- [ ] Add the PyPI token as a GitHub Actions secret:
  - Go to Settings → Secrets and variables → Actions → New repository secret
  - Name: `PYPI_API_TOKEN`
  - Value: Your PyPI token
  - Ensure the secret is available to workflows

### 3. Branch Protection (Recommended)
- [ ] Go to Settings → Branches → Add rule for `main`
- [ ] Enable "Require status checks to pass before merging"
- [ ] Require these workflows:
  - `test`
  - `commit-lint`
- [ ] Enable "Require branches to be up to date before merging"

### 4. Pre-commit Installation (Local Development)
- [ ] Install pre-commit hooks:
  ```bash
  pip install pre-commit
  pre-commit install --hook-type commit-msg --hook-type pre-commit
  ```
- [ ] Test the hooks work:
  ```bash
  # This should fail (no type)
  git commit --allow-empty -m "test commit"

  # This should pass
  git commit --allow-empty -m "chore: test commit"
  ```

### 5. Release Immutability (Required for Maximum Security)

GitHub's immutable releases feature prevents release tampering. **Enable it:**

1. Go to repository Settings
2. Scroll to "Releases" section
3. Check **"Enable release immutability"**
4. Apply and note: "immutability will only apply to future releases"

**What this does:**
- ✅ Git tags become immutable (cannot be moved, changed, or reused)
- ✅ Release assets cannot be modified or deleted
- ✅ Automatic release attestation for verification
- ✅ Prevents supply chain attacks

**Once enabled, releases follow this workflow:**
1. Create release as **draft** (assets can still be added)
2. Attach all files/assets
3. Publish to make immutable (final and permanent)

### Code Quality
- [ ] All tests pass locally: `uv run pytest`
- [ ] Linting passes: `uv run ruff check src/ tests/`
- [ ] Code is formatted: `uv run ruff format src/ tests/`
- [ ] No unresolved review comments or TODOs in critical code

- [ ] API documentation is current (if applicable)

  - Correct classifiers

- [ ] No API credentials hardcoded anywhere
- [ ] Package builds cleanly: `python -m build`
- [ ] Built wheel can be installed: `pip install dist/*.whl`

## Release Process

Once setup is complete, the release workflow is **automatic**:

1. **Make changes** on a feature branch
2. **Open PR** and wait for tests to pass
3. **Merge to main** with a conventional commit message (see below)
4. **GitHub Actions automatically**:
   - Analyzes commit history
   - Bumps version (major/minor/patch)
   - Updates `CHANGELOG.md`
   - Tags release
   - Publishes to PyPI

### Conventional Commit Messages

The release workflow uses commit messages to determine version bumps. Use these formats:

- `fix: description` → Patch bump (0.1.0 → 0.1.1)
- `feat: description` → Minor bump (0.1.0 → 0.2.0)
- `feat!: description` or `BREAKING CHANGE:` → Major bump (0.1.0 → 1.0.0)
- `docs:`, `chore:`, `style:`, `refactor:`, `test:` → No version bump

Pre-commit hooks will automatically validate commit messages locally before pushing.

Example:
```
feat: add colour temperature control

- Implement set_colour tool
- Support Kelvin temperatures
- Add RGBWAF support

Closes #123
```

**Valid scopes** (optional but recommended):
- `auth` — Authentication/OAuth
- `api` — API client
- `tools` — MCP tools
- `models` — Data models
- `scope` — Scope control
- `resources` — Resource discovery
- `dev` — Development utilities
- `docs` — Documentation
- `chore` — Maintenance
- `ci` — CI/CD workflows

## Troubleshooting

### Workflow doesn't trigger
- Verify `PYPI_API_TOKEN` secret is set in repository
- Check branch protection rules don't block the workflow
- Ensure commits follow conventional commit format

### PyPI token issues
- Token expired? Create a new one and update the secret
- Wrong scope? Ensure token has "Entire repository" scope

### Version didn't bump
- Check that commit messages follow conventional commit format
- Ensure previous releases used semantic versioning
- Run `uv run semantic-release version --no-push` locally to debug
